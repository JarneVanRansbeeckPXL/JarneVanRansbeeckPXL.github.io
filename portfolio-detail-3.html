Het seminarie die mij het meest bijgebleven is, is een seminarie van Eurofins over het onderwerp ‘Test Driven Development’. Hierbij gaf gastspreker Martijn Degreve een uitleg over Eurofins zelf gevolgd door de waarde die Test Driven Development binnen Eurofins zelf heeft. Vervolgens ging hij dieper in op dit topic gevolgd door een praktijkgedeelte.

TDD (Test Driven Development) heeft drie grote regels. Deze regels zijn: - Je kan geen productiecode schrijven tot je een gefaalde unit test hebt geschreven - Schrijf niet meer code van een unit test dan nodig is om te falen en niet compileren telt ook als falen - Je kan niet meer productiecode schrijven dan nog is om een huidige falende unit test te laten lukken.

Rond deze drie regels is er een loop gecreëerd. Deze loop heeft drie verschillende stages: schrijf een test die faalt, zorg ervoor dat de test wel lukt, schoon de code op.

Deze loop moet herhaald worden voor elke test die geschreven wordt. Dit proces zorgt voor een goede manier van werken waarbij er enorm veel tijdswinst behaald wordt op het uiteindelijke resultaat.

Het praktijkgedeelte ging over de klassieker ‘FizzBuzz’. Dit probleem hebben we bij het opleidingsonderdeel .NET moeten maken voor onze permanente evaluatie. Tijdens deze seminarie hebben we dit op een andere manier aangepakt. Namelijk door gebruik te maken van TDD en ons hier ook aan te houden.

De opdracht van FizzBuzz is om een programma te schrijven waarbij de nummers 1 tot 100 geprint worden. Wanneer er een tafel van drie is moet er Fizz geprint worden in plaats van het nummer en wanneer het een tafel van vijf is het woord Buzz. Voor nummers die zowel tafel van drie als van vijf zijn moet het woord FizzBuzz geprint worden.

Om de opdracht uit te voeren volgens de manier van TDD hebben we eerst een test geschreven die bij nummer “3” het woord “Fizz” verwacht. Uiteraard faalt deze test, want dat is de bedoeling van TDD. Vervolgens zorgen we ervoor dat we code schrijven die deze test laat lukken. Vervolgens kunnen we de code opschonen en er voor zorgen dat niet enkel het nummertje drie getest wordt, maar alle tafels van drie.

Dit proces herhalen we voor het nummer “5”, waarbij de output “Buzz” verwacht wordt. Eenmaal dit gelukt is, kunnen we dit ook opschonen en zorgen dat het werkt bij meerdere use cases.

Dit doen we voor alle nodige testjes totdat ons programma werkt zoals bedoeld is. Vervolgens kunnen de tests alsnog verbeterd worden om zo efficiënt mogelijk te werken.

Binnen TDD zijn er natuurlijk meerdere mogelijkheden voor een opdracht aan te pakken. Hierdoor kregen we op het einde van het seminarie enorm verschillende oplossingen. Sommige van deze oplossingen waren totaal niet efficiënt en hadden meerdere testjes nodig voor 1 use case te testen, terwijl andere mensen dit in 1 test konden doen. Ikzelf had een niet zo’n geoptimaliseerde oplossing, maar dat was te verwachten doordat ik niet meer veel bezig ben met geweest met programmeren.

Ik heb deelgenomen aan deze seminarie omdat ik geïnteresseerd ben in de verschillende soorten manieren voor een probleem op te lossen. Test Driven Deployment is zo’n aanpak waar ik nog nooit van had gehoord, maar wel mijn interesse opwekte. Het concept van deze aanpak is vrij simpel, maar lijkt op het eerste zicht niet aangenaam. Het vruchtbare resultaat dat achteraf zichtbaar is, toont wel aan dat deze methode van werken het waard is.

Omdat ik systeem- en netwerkbeheer studeer, heb ik al een tijdje niet meer zitten programmeren. Deze seminarie had wel de vereiste om te kunnen programmeren in een bepaalde taal. Dit was niet een groot probleem, maar zorgde er wel voor dat ik mijn tijd nodig had. Mijn struikelblokken werden meteen verholpen door de gastspreker zelf terwijl het seminarie verliep via Teams door de huidige omstandigheden. Deze ervaring had ik niet met andere seminaries die online waren. Ook waren de andere junior-collega’s een enorme hulp. Het was natuurlijk ideaal dat de gekozen opdracht al eerder bij .NET aan bod was gekomen waardoor het schrijven van de code zelf niet zo moeilijk was.

Ik zal waarschijnlijk niet meteen in aanraking komen met deze methode na deze seminarie, maar ik hecht wel belang aan kennis op te doen op alle vlakken van informatica.

De reden dat ik deze seminarie specifiek selecteren uit de hele lijst die ik opgesteld heb, is omdat de gastspreker op een korte tijdsperiode zijn onderwerp naar ons toe heeft kunnen brengen. Dit heeft hij gedaan op een interessante manier waarbij mijn aandacht nooit verloren geraakte. Ik herinner mij deze seminarie waarschijnlijk nog jaren later doordat Martijn dit zo goed heeft uitgelegd.
